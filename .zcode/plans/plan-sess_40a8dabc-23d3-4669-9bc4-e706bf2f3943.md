## Rediseño de la UI de ADO: wizard guiado + gestión compacta

### Objetivo
Reemplazar la página `/ado` densa (4 bloques que repiten `org` y `patEnvVar` por cada profile) por **(a) un wizard guiado** paso a paso (como `ado init`) abierto desde un botón, y **(b) una vista de gestión compacta**. La `org` se pide **una vez** y se comparte; `patEnvVar` desaparece de la UI (siempre `AZURE_DEVOPS_PAT`); el nombre del profile se **auto-genera** del `project` (slug).

### Modelo de wizard (5 pasos, en `<Modal width="960px">`)
Patrón copiado de `CreateMissionModal.tsx` (`STEPS[]` + `step` state + step indicator + footer Back/Next).

```
① Organization ──② PAT ──③ Profiles ──④ Project rules ──⑤ Review
   ▰▰                                            ░░░░░░

Step 1 — Organization (una sola vez)
  Organization URL or name:  [ myorg              ]
  e.g. myorg, dev.azure.com/myorg — shared by all profiles

Step 2 — PAT
  ✓ PAT already configured (~/.azure-devops-cli/pat)   [Skip ▸]
    — o —
  Paste your PAT:  [ •••••••••• ]
  Scopes: Code (R/W), PR Contribute (R/W), Work Items (Read)
  [collapsible: cómo setear AZURE_DEVOPS_PAT como env var — compactado]

Step 3 — Profiles (loop, sin org, sin patEnvVar)
  ┌ Default  ─────────────────────────────────┐
  │ Project: [ WebPlatform        ]            │
  │ Repos:   [web-api] [frontend] [+ add]      │
  └────────────────────────────────────────────┘
  ┌ ──────────────────────────────────────────┐
  │ Project: [                  ]   ✕          │
  │ ...                                       │
  └────────────────────────────────────────────┘
  [+ Add project]        (nombre auto-generado del project)

Step 4 — Project rules (.adoconfig.toml) [opcional]
  [✓] Generate .adoconfig.toml
  Strategy: [feature-chain ▾]  Base branch: [main]
  ... (formulario actual, compactado)
  ┌ Run in your repo ──────────────  [Copy] ┐
  │ cat > .adoconfig.toml <<'EOF' ...        │
  └──────────────────────────────────────────┘

Step 5 — Review
  Organization: myorg
  PAT: ✓ stored
  Profiles: webplatform (default), backend-api
  .adoconfig.toml: will be generated (copy)
                          [Back]  [Apply ✓]
```

Al **Apply**: `POST /api/ado/pat` (si se ingresó PAT) → `POST /api/ado/config` (todos los profiles con la misma `org` y `patEnvVar:"AZURE_DEVOPS_PAT"`). Cierra el modal y refresca. Sin endpoints nuevos.

### Vista de gestión compacta (la página `/ado` cuando ya hay config)
```
┌─ Azure DevOps ──────── ado v0.8.11 ✓ ──────────────┐
│  ✓ CLI   ✓ PAT   ✓ 2 profiles      [🪄 Guided setup]│
│  Organization: myorg                                 │
├──────────────────────────────────────────────────────┤
│  Profiles                                            │
│  ┌─────────────────────────┐ ┌─────────────────────┐ │
│  │ webplatform  [Default]  │ │ backend-api         │ │
│  │ repos: web-api, frontend│ │ repos: api          │ │
│  │ [Edit] [Delete]         │ │ [Edit] [Delete]     │ │
│  └─────────────────────────┘ └─────────────────────┘ │
│  [+ Add profile]                                     │
│                                                      │
│  [Build .adoconfig.toml ▾]   (collapsible, igual que hoy)
└──────────────────────────────────────────────────────┘
```
- La `org` se muestra **una vez** como encabezado global (derivable del profile default). Las cards ya no repiten org ni patEnvVar.
- Modal Add/Edit de gestión: solo `project` + `repos` (org heredada del default).
- Tutorial de env var: se compacta a un collapsible dentro del modal PAT (no 3 bloques en la página).
- Builder `.toml`: queda accesible como collapsible en gestión **y** como paso 4 del wizard.

- Prueba manual: abrir `http://localhost:5768/ado`, recorrer el wizard (incluyendo "ya hay PAT → Skip"), verificar que guarda `~/.config/opencode/ado.json` con la misma `org` en todos los profiles y `patEnvVar: "AZURE_DEVOPS_PAT"`, y que la vista de gestión refleja los cambios sin campos repetidos.

### Notas / decisiones ya tomadas
- **No** se añade endpoint nuevo: se reusan los existentes (mantiene el backend simple).
- **No** se cambia el modelo de datos `AdoProfile` (sigue teniendo `org` y `patEnvVar` por profile por compatibilidad con el CLI); solo la UI deja de pedirlos repetidamente.
- Profiles con datos legacy (`org` distintas por profile) siguen funcionando: la vista de gestión deriva la "org global" del profile default; si hay heterogeneidad se muestra un aviso.
- El nombre del profile se auto-genera del `project` (slug) como hace `ado init`; no se pide manualmente en el wizard. En el modal de gestión se permite renombrar al editar.
