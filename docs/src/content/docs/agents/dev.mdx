# Dev Agent

## Qué hace

Implementa features, arregla bugs, y refactoriza código.

## Cuándo usarlo

| Tarea | Ejemplo |
|---|---|
| Implementar feature | `@dev Implementa login con JWT` |
| Arreglar bug | `@dev El endpoint /api/users retorna 500` |
| Refactorizar | `@dev Refactoriza UserService con repository pattern` |
| Agregar validación | `@dev Agrega validación al formulario de registro` |

## Flujo de trabajo

1. **Lee** archivos relacionados para entender el contexto
2. **Planifica** los cambios
3. **Implementa** cambios atómicos
4. **Verifica** con tests, lint, type-check
5. **Limpia** código muerto, TODOs, debug statements

## Principios

| Siempre | Nunca |
|---|---|
| Usar tipos existentes | Dejar `console.log` de debug |
| Manejar errores explícitamente | Agregar `// TODO` sin issue |
| Funciones pequeñas | Romper tests existentes |
| Nombres descriptivos | Introducir dependencias sin verificar |

## Handoff

Cuando termina, reporta:
```
Status: done | blocked | needs-decision
Did: <resumen>
Artifacts: <archivos cambiados>
Next: @dev | @qa | @reviewer | @devops | close
```

## Límites

| ✅ Puede | ❌ No puede |
|---|---|
| Leer, escribir, editar código | Decisiones de arquitectura (`@architect`) |
| Correr tests y build | Revisar su propio código (`@reviewer`) |
| Debuggear y arreglar | Diseñar estrategia de testing (`@qa`) |

<Aside type="tip">
Sé específico: en vez de "arreglá el bug", decí "el endpoint POST /api/login retorna 500 cuando el email tiene '+'".
</Aside>
