# Architect Agent

El agente architect se especializa en **diseño de software, patrones arquitectónicos y decisiones técnicas**. Evalúa tradeoffs, propone soluciones y documenta decisiones.

<Badge text="Rol: Arquitecto" variant="success" />

## Cuándo usarlo

| Tarea | Ejemplo |
|---|---|
| Decisiones de diseño | "Should we use event sourcing for this?" |
| Evaluación de patrones | "Is CQRS appropriate here?" |
| Análisis de tradeoffs | "Monolith vs microservices for this feature" |
| Revisión de arquitectura | "Review the current auth architecture" |
| Planificación de refactoring | "How should we restructure the data layer?" |

## Lo que hace

1. **Evalúa** el código existente y la estructura del proyecto
2. **Identifica** oportunidades de mejora arquitectónica
3. **Propone** soluciones con tradeoffs claros
4. **Documenta** decisiones (ADRs cuando aplica)
5. **Delega** implementación a `@dev`

## Lo que NO hace

- ❌ NO implementa código (eso es `@dev`)
- ❌ NO revisa PRs por estilo (eso es `@reviewer`)
- ❌ NO configura infraestructura (eso es `@devops`)
- ❌ NO escribe tests (eso es `@qa`)

## Ejemplo de uso

```
@architect We need to add real-time notifications to the app.
Current stack is Next.js + PostgreSQL. Should we use WebSockets,
SSE, or a service like Pusher? Consider scalability, complexity,
and our team size.
```

<Aside type="tip">
El agente architect es ideal antes de empezar features grandes. Úsalo para tomar decisiones informadas antes de que `@dev` implemente.
</Aside>

## Flujo típico

```
@architect (diseña) → @dev (implementa) → @qa (testea) → @reviewer (revisa)
```

## Documentación que genera

- **ADRs** (Architecture Decision Records) para decisiones importantes
- **Diagramas** de arquitectura cuando es necesario
- **Análisis de tradeoffs** con pros/cons de cada opción
