# Agentes

ywai te da acceso a **agentes especializados** que trabajan como un equipo. Cada agente tiene un rol claro y sabe exactamente qué puede y qué no puede hacer.

## El equipo

| Agente | Rol | Para qué sirve |
|---|---|---|
| [**orchestrator**](/dev-ai-workflow/agents/orchestrator) | Coordinador | Coordina múltiples agentes para tareas complejas |
| [**dev**](/dev-ai-workflow/agents/dev) | Desarrollador | Escribir código, arreglar bugs, refactorizar |
| [**qa**](/dev-ai-workflow/agents/qa) | QA Engineer | Tests, cobertura, validación |
| [**architect**](/dev-ai-workflow/agents/architect) | Arquitecto | Decisiones de diseño, patrones, tradeoffs |
| [**reviewer**](/dev-ai-workflow/agents/reviewer) | Revisor | Code review, bugs, seguridad |
| [**devops**](/dev-ai-workflow/agents/devops) | DevOps | CI/CD, Docker, K8s, despliegues |
| [**ask**](/dev-ai-workflow/agents/ask) | Investigador | Preguntas, análisis, explicaciones |
| [**finder**](/dev-ai-workflow/agents/finder) | Explorador | Buscar archivos, entender estructura |
| [**planning**](/dev-ai-workflow/agents/planning) | Planificador | Investigar y planificar antes de implementar |
| [**memory**](/dev-ai-workflow/agents/memory) | Archivista | Memoria persistente entre sesiones |

## Cómo usarlos

En opencode, usa `@nombre-del-agente` seguido de tu tarea:

```
@dev Implementa el endpoint de login con JWT
@qa Escribe tests para el servicio de autenticación
@architect ¿Qué patrón usamos para el manejo de estado?
@reviewer Revisa el PR del feature de pagos
@devops Crea un Dockerfile para el servicio de Node.js
@ask Cómo funciona el middleware de auth actual?
@finder Dónde está la lógica de autenticación?
@planning Planifica la implementación de RBAC
@memory Qué decidimos sobre la estrategia de migración?
```

## Flujo natural del equipo

Los agentes están diseñados para trabajar juntos. Un flujo típico:

```
@finder (explora) → @ask (investiga) → @planning (planifica) → @architect (diseña) → @dev (implementa) → @qa (testea) → @reviewer (revisa) → @devops (despliega)
```

No tienes que seguir este orden exacto — úsalos según necesites.

## Ejemplos prácticos

### Feature nueva

```
Tú: "@architect Necesitamos agregar notificaciones push. 
     ¿Qué arquitectura recomiendas?"

@architect: [analiza opciones, propone diseño]

Tú: "@dev Implementa según el diseño de @architect"

@dev: [escribe el código]

Tú: "@qa Cubre con tests el nuevo servicio de notificaciones"

@qa: [escribe tests]

Tú: "@reviewer Revisa antes de mergear"

@reviewer: [encuentra 2 issues, sugiere fixes]

Tú: "@dev Arregla los issues encontrados por @reviewer"

@dev: [corrige]

Tú: "Listo para merge"
```

### Bug rápido

```
Tú: "@dev Los usuarios no pueden hacer login con Google, 
     el callback falla con 500"

@dev: [investiga, encuentra el bug, arregla]

Tú: "@qa Agrega test para este caso"

@qa: [escribe test que previene regresión]
```

### Pregunta técnica

```
Tú: "@ask Cuál es la diferencia entre SSR y SSG en Next.js 
     para nuestro caso de uso?"

@ask: [explica con contexto del proyecto]
```

## Cuándo usar cada agente

<Card title="Guía rápida" icon="star">

**Necesitas código** → `@dev`

**Decisiones de diseño** → `@architect`  
**Revisar código** → `@reviewer`  
**Infraestructura** → `@devops`  
**Buscar código** → `@finder`  
**Planificar antes de implementar** → `@planning`

</Card>

## Agentes personalizados

Puedes crear tus propios agentes o modificar los existentes. Cada agente tiene:

- **`AGENT.md`** — El system prompt (instrucciones del agente)
- **`tools.json`** — Qué herramientas puede usar
- **`skills.txt`** — Qué skills tiene vinculados

Los perfiles están en `ywai/agents/`. Puedes copiar la estructura y ajustar según las necesidades de tu proyecto.

<Aside type="tip">
Los agentes funcionan mejor cuando les das contexto específico. En vez de "arregla el bug", di "el endpoint `/api/login` retorna 500 cuando el email tiene caracteres especiales".
</Aside>
