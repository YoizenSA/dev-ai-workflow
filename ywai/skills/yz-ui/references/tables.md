# Tables, lists, dashboards & pagination — field-tested lessons

Read this when building any data table, list, dashboard card or paginated view.
The base classes (`.data-table`, `.table-wrap`, `.col-hide-*`, `.cell-trunc`,
`.cell-stack`, `.cell-sub`, `.col-fit`, `.mini-empty`, `.card-foot-link`,
`.kpi-compact`) all ship in `table.css`/`components.css`.

## Cells, truncation & hover

- **Custom list rows need a row-level hover, not just brighter actions.** `.data-table` rows get a hover background for free, but hand-rolled lists (accordion bodies, version/item rows) often only reveal/brighten the trailing `.icon-act` on `:hover` — that's not enough feedback to tell *which* row you're on, and it fails in both themes. Give the whole row a `:hover { background: var(--surface-hover); }` (rounded, with a `transition`) on top of the action reveal. The token flips per theme, so one rule covers dark and light.
- **Truncation**: `.cell-trunc` (`max-width + ellipsis + display:block`) goes on an inner `<span>`, NEVER on the `<td>` — a block `td` leaves table layout and its row border misaligns with siblings. When the same table renders in more than one place (compact card + "view all" modal, list + detail), every copy must repeat the exact truncation/`col-fit` markup — any copy without it is the one that meets the longest value.
- **Externally-sourced text truncates by default, everywhere it appears.** Apply `.id-trunc` from day one to any column whose values the team doesn't control — user-entered names, identifiers, free-form targets (`<span class="cell-mono id-trunc" [attr.data-tip]="val.length > 28 ? val : null">`). Defined once in `table.css`, never per-component. Skip it for short, team-controlled vocabulary (catalog/enum names): truncating controlled text is noise.
- **Pill columns** (any enum rendered as a pill: status, environment, action type): center both `th` and `td` (`.col-c`/`.col-fit`); text columns stay left.
- **Status cells**: keep the pill alone; secondary detail (error message, author) goes in a tooltip or a `.cell-sub` line — never wrapped text under a centered pill.

## Containment & responsive columns

- **A table wider than its content area must scroll inside its wrapper — never `overflow: visible` on `.table-wrap`.** The default `.table-wrap { overflow-x: auto }` is load-bearing: it contains a too-wide table to its card. Overriding to `visible` (a tempting fix so cell tooltips don't clip) lets the table paint past its container and push the **whole page** wide on narrow viewports (the sidebar eats ~264px) — the entire app scrolls horizontally, not just the table. You cannot have both "tooltips never clip" (needs `visible`) and "page never breaks" (needs a scroll container) when the table doesn't fit. Containment wins. The clipped edge-tooltip is the accepted cost.
- **Then reduce *how often* it scrolls by hiding low-priority columns at breakpoints** — a complement to containment, not a substitute. `.col-hide-lg`/`.col-hide-md` on both `th` and `td` → `display:none` under 1300/1040px. Keep a never-hidden core (identifier + 2-3 status columns + actions); drop detail columns reachable elsewhere (paths, repo/branch live in the edit view).
- **Table column hierarchy**: the header must visibly differ from the body or the grid stops reading. `thead` gets its own background token (`--table-head-bg`, distinct from body rows) + a `--panel-border-strong` bottom border; keep row separators faint but present (~0.10-0.12 alpha). On a light theme an almost-white header over white rows disappears — that's the tell.

## Filters

- **Filter labels live *inside* the control**: in a filter bar, a title stacked above each control (or beside it) reads as a load-form and adds noise. Embed the label in the control instead (the `label` input on `yd-select`/`yd-date` renders "Usuario: Todos ▾", "Desde: dd/mm/aaaa"). Stacked uppercase labels are for *edit* forms (modals, editors), not filter bars.
- **Filter a log/list by a related entity via deep-link, not a giant dropdown**: when one view (audit, history, events) is naturally scoped by an entity that lives in another, paginated view (clients, users), don't stuff a non-searchable `yd-select` with hundreds of options. Add a row action in the source view ("Ver auditoría") that navigates with the id as a query param (`[queryParams]="{ clientId: id, client: identifier }"`), and have the target read it on init (`route.snapshot.queryParamMap`), apply it as a filter, and render it as a **removable active-filter chip** (entity label + "Quitar filtro" ✕). Drive the chip off the filter signal, not the URL. Gate the source action by the same role the target route requires. Pass the human-readable identifier as a second display-only param so the chip shows a name, not a raw ObjectId.

## Pagination

- **Every report/list paginates at 10 rows** with a working `Anterior · N / M · Siguiente` pager in `.table-foot` ("X–Y de Z"). Never render a dead/disabled pager or dump the full dataset.
- **Pagination must be server-side when the endpoint supports it** (think 800+ rows, not the 5 in dev): pass `page/limit` + filters to the API, keep `rows`/`total` signals, debounce search 300ms resetting to page 0, re-query on any filter/page change (signals `effect`), and step back a page when a delete empties the last one. **The effect must call the loader inside `untracked()`**: the loader's sync part reads derived signals (`safePage()` → `total`) that the loader itself writes, so without `untracked` the effect subscribes to its own output and re-fires on every response (~1 req/sec runaway loop). Client-side slice is only acceptable for small in-memory sets (e.g. a dashboard "ver todos" modal). Clamp: `safePage = min(page, pageCount-1)`.

## Dashboards & cards

- **Dashboards summarize, reports paginate**: overview cards show top-5 with no pager. When the source holds more items than shown, the pinned footer becomes a `.card-foot-link` "Ver los N →" opening a modal with the full table (reuse the data already in memory — don't navigate to a screen that can't reproduce the filter, and don't duplicate backend business rules client-side).
- **Paired dashboard cards must look right with unequal data**: `.grid-equal .card { display:flex; flex-direction:column }`, the `.table-wrap` takes `flex:1`, and a pinned `.card-foot` summary line sits at `margin-top:auto`. Both cards stay same height and the shorter one reads intentional, not empty.
- **Copy must degrade with low/zero data**: a "Top 5 …" heading over 1 row reads absurd — switch by count (show "Top 5 …" only when the list is truncated; "N en uso" when it's short; a "Sin datos" line at zero). Empty tables inside cards get a compact centered `.mini-empty` row (icon + sentence; `.ok` green check for good-news empties like "todo al día"), never a bare header over blank space.
- **Pinned footer copy must be honest about truncation**: if the API caps the list (top-N), label it as a top-N ("Top 5 con más …") — never compute totals from a truncated payload. When there IS a full count, the link copy is "Ver todos (N) →".
- **Interactive elements flush with rounded card edges need matching radius**: a hover background on a bottom-pinned footer-link paints square corners outside the card — give it `border-radius: 0 0 var(--radius-lg) var(--radius-lg)` (already in `.card-foot-link`).
- **KPI cards are compact and horizontal** (`.kpi-compact`): icon box at the left, value+label stacked beside it, tight padding. The stacked icon-above-number layout wastes half the card and looks empty at desktop widths.
- **Bars in tables**: cap width (`flex: 0 1 320px`), inline muted `%` next to the bar, plain `tnum` count in its own narrow column — don't repeat the column header's word inside the cell, and never let a full-width bar leave a lone number floating at the far edge.

## Grouped catalogs & big blobs

- **Grouped catalogs collapse**: when a list groups N items under a parent (app → versions, component → fields), render each group as an accordion **collapsed by default** with an info-rich header (name, "N versiones · M activas", latest `.tag`, rotating chevron). An active search auto-expands matches and disables manual toggling. Never render all groups expanded.
- **Inside an expanded group, cap the list too**: show the 10 most recent items + an incremental "Mostrar 10 más (N restantes)" button. Versions/history skew to "latest matters". Active search bypasses the cap. Once expanded past the initial batch, always offer "Mostrar menos". Group lists get an Estado (Todas/Activas/Inactivas) `.filter-inline` next to the search; an active filter auto-expands matching groups like search does — but auto-expand is a default, not a lock: the user can still collapse groups manually while filtering.
- **Rarely-edited blobs (manifests, certs, global templates)**: don't drop a giant inline `<textarea>` into the page. Show a compact summary row (`name · N líneas · meta`) with **Copiar** (clipboard, no need to open it) and **Ver / Editar** that opens a modal with the full editor (~860px, textarea ~46vh) + placeholder chips + a copy-all action.

## Ordering, naming & pill mapping

- **One noun per entity, everywhere**: whatever the nav item calls an entity, the KPIs, card titles, table headers and modals use that same word — never alternate synonyms across the same screen.
- **Card titles must describe the actual ordering**: "más antiguas" implies sorted by date; if the API orders by something else, verify the service. Preference: **respect the backend's intended ordering** (it usually encodes a business rule) and make the UI honest about it — accurate title + one muted description line under the section head explaining the criterion. Re-sorting client-side to match a wrong title is the fallback, not the default.
- **Map a domain value → pill variant with an explicit `Record`, never substring-matching the raw key.** `{ prd:'danger', stg:'warning', … }` is precise and scannable; `if (action.includes('delete')) …` is fragile. Two hard rules: (1) the variant string **must** be a real `.pill-<x>` class — a typo'd/legacy variant renders a **colorless** pill with no warning, so type it (`PillVariant` union) and let the compiler catch it; (2) for open-ended keys (audit actions) keep an explicit map for the known set **plus** a verb-based fallback whose default is a neutral `info`/`muted`, not an alarming colour. Choose hues by meaning: red = real *failures*, amber = intentional *destructive* (delete), blue = edits, green = created/deployed, grey = low-signal (login/view).
- **Audit/detail modals show real data, never decorative diffs**: render `changes[]` as rows (`field` mono · old struck-through red → new green; secrets as `••••••••`), flat metadata as `.tag` chips, and an honest empty state when there's nothing. A fake hardcoded diff is a bug.
